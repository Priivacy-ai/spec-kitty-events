### Implementation Notes

- Update `_EVENT_TO_STATE` mapping: `USER_CONNECTED` → `ConnectorState.PROVISIONED`, `USER_DISCONNECTED` → `ConnectorState.REVOKED`.
- Update `_EVENT_TO_PAYLOAD` mapping with new payload types.
- Update `ConnectorPayload` union type to include new payloads.

## Work Package WP02: Reducer & Roster Logic (Priority: P0)

**Goal**: Extend `ReducedConnectorState` with `user_connections` roster and update `reduce_connector_events()` to build per-user state from `user_id` fields.
**Independent Test**: Reducer produces correct roster for streams with/without user_id, including mixed binding-level and user-level events. Backward compatibility verified with pre-migration event streams.
**Prompt**: `tasks/WP02-reducer-and-roster.md`
**Requirement Refs**: FR-005, FR-007, NFR-001, C-002
**Estimated Prompt Size**: ~300 lines

### Included Subtasks
- [x] T006 Add `user_connections: Tuple[UserConnectionStatus, ...] = ()` to `ReducedConnectorState` in `src/spec_kitty_events/connector.py`
- [x] T007 Update `reduce_connector_events()` to build per-user roster from `user_id` on binding-level events in `src/spec_kitty_events/connector.py`
- [x] T008 Handle `UserConnected`/`UserDisconnected` in reducer: skip binding-level state transitions, update roster only in `src/spec_kitty_events/connector.py`
- [x] T009 Add anomaly handling for `UserDisconnected` without prior `UserConnected` in `src/spec_kitty_events/connector.py`

### Implementation Notes
- Mutable roster dict `Dict[str, Tuple[ConnectorState, Optional[datetime]]]` keyed by `user_id` during fold, frozen to `Tuple[UserConnectionStatus, ...]` at output.
- For binding-level events with `user_id` set: update roster with the event's target state and `recorded_at`.
- For `UserConnected`/`UserDisconnected`: update roster but skip `current_state`/`transition_log` (binding-level state unaffected).
- Events without `user_id` (legacy): no roster update, binding-level processing unchanged.
- Sort roster by `user_id` in output for determinism.
- `UserDisconnected` for unknown user: record anomaly (kind="invalid_transition").

### Parallel Opportunities
- T006 (model change) can proceed before T007-T009 (logic changes).

### Dependencies
- Depends on WP01 (new models, constants, and event type mappings must exist).

### Risks & Mitigations
- Determinism: roster must be sorted by `user_id` to ensure same output regardless of event ordering.
- Backward compat: verify with event stream containing zero `user_id` fields — roster must be empty, binding-level state identical to pre-change.
