# WP05 — Conformance fixtures (valid + invalid)

## Objective

Ship committed valid and invalid conformance fixtures for every V1 payload shape and wire the existing conformance test to iterate them. Lock NFR-002 at the file-system level.

## Context

The repo's conformance convention is to commit per-payload fixtures under `src/spec_kitty_events/conformance/fixtures/<domain>/{valid,invalid}/`. `tests/test_decisionpoint_conformance.py` iterates these and validates them against both the committed JSON Schemas and the Pydantic model classes.

See:
- `data-model.md` §8.1 and §8.2 — exhaustive fixture list.
- `spec.md` FR-016, FR-017, NFR-002 — requirements.
- Existing fixtures under `src/spec_kitty_events/conformance/fixtures/decisionpoint/valid/` for envelope-shape reference.

## Branch strategy

- Planning/base branch: `main`
- Final merge target: `main`
- Execution worktree: assigned per lane; runs after WP02 and WP04 complete.

## Subtasks

### T019 — Add valid conformance fixtures

**Purpose.** Prove every V1 payload shape validates against both the JSON Schema and the Pydantic model.

**Steps.**

1. Inspect an existing valid fixture (e.g., `conformance/fixtures/decisionpoint/valid/*.json`) to learn the envelope shape used by `test_decisionpoint_conformance.py`. Typical shape:
   ```json
   {
     "event_id": "01JA...",
     "event_type": "DecisionPointOpened",
     "actor_id": "...",
     "tenant_id": "...",
     "lamport": 1,
     "timestamp": "2026-04-23T10:00:00+00:00",
     "payload": { ... typed payload dict ... }
   }
   ```
2. For each of the 10 valid fixtures listed in `data-model.md` §8.1, write a single-event fixture. Use deterministic ULIDs (e.g., `01J10000000000000000000001` incremented per fixture) so bytes are stable. Use timestamps in the form `2026-04-23T10:00:00+00:00`.
3. Specific fixtures:
   - `v1_opened_adr.json` — `origin_surface="adr"`, full ADR fields (rationale="Legacy ADR", alternatives_considered=["a","b"], evidence_refs=["e1"], authority_role="mission_owner", authority flag True, etc.).
   - `v1_opened_interview.json` — `origin_surface="planning_interview"`, `origin_flow="specify"`, question, options=["sessions","oauth2","oidc","Other"], input_key, step_id, actor_type="human".
   - `v1_widened.json` — channel="slack", teamspace_ref, default_channel_ref, thread_ref with url, invited_participants=2 entries each with `external_refs.teamspace_member_id`, widened_by=ULID.
   - `v1_discussing_interview.json` — snapshot_kind="digest", contributions=2 strings, actor_type="service".
   - `v1_resolved_interview_resolved.json` — terminal_outcome="resolved", final_answer="oauth2", other_answer=false, summary={text,source="slack_extraction",extracted_at,candidate_answer}, actual_participants=2 entries, resolved_by=ULID, closed_locally_while_widened=false, closure_message populated.
   - `v1_resolved_interview_resolved_other.json` — terminal_outcome="resolved", final_answer="hand-rolled SSO plugin", other_answer=true, summary={text,source="mission_owner_override"}.
   - `v1_resolved_interview_deferred.json` — terminal_outcome="deferred", rationale="need security review", no final_answer, no other_answer, no summary.
   - `v1_resolved_interview_canceled.json` — terminal_outcome="canceled", rationale="out of scope", no final_answer.
   - `v1_resolved_interview_closed_locally.json` — terminal_outcome="resolved", final_answer="session-based", closed_locally_while_widened=true, summary={text,source="mission_owner_override"}, closure_message populated, actual_participants may be empty.
   - `v1_participant_identity_with_external_refs.json` — shared fixture demonstrating a `ParticipantIdentity` with all three `external_refs` populated. Since this is a standalone value, wrap it in a thin envelope or store as a raw JSON object; consult how `tests/test_decisionpoint_conformance.py` handles non-event fixtures.

**Files.** 10 new JSON files under `conformance/fixtures/decisionpoint/valid/`.

**Validation.**
- [ ] Each fixture parses as JSON without errors.
- [ ] Each fixture validates against the matching committed JSON Schema via `jsonschema.Draft202012Validator`.
- [ ] Each fixture validates against the matching Pydantic model via `TypeAdapter(...).validate_python(...)`.

### T020 — Add invalid conformance fixtures

**Purpose.** Lock negative schema enforcement so silent regressions are impossible.

**Steps.**

1. Write each fixture listed in `data-model.md` §8.2. Each must be *minimally* invalid — omit or mis-set exactly one field so the failure point is obvious.
   - `v1_resolved_missing_terminal_outcome.json` — interview-variant Resolved envelope missing `terminal_outcome`. Schema must reject.
   - `v1_widened_missing_thread_ref.json` — Widened envelope missing `thread_ref`.
   - `v1_opened_interview_missing_origin_flow.json` — interview Opened missing `origin_flow`.
   - `v1_participant_identity_empty_external_refs.json` — `external_refs: {}` with no fields set.
   - `v1_resolved_interview_deferred_with_final_answer.json` — `terminal_outcome: "deferred"` AND `final_answer: "wrong"` — violates cross-field rule.
2. Each invalid fixture must include a `"__comment": "..."` field explaining why it's invalid (optional but strongly recommended — the test can ignore it, but reviewers will thank you).

**Files.** 5 new JSON files under `conformance/fixtures/decisionpoint/invalid/`.

**Validation.**
- [ ] Each fixture parses as JSON.
- [ ] Each fixture FAILS schema validation (for the first four) and cross-field Pydantic validation (for the fifth).

### T021 — Extend `tests/test_decisionpoint_conformance.py`

**Purpose.** Make the test suite iterate the new fixtures automatically.

**Steps.**

1. Open `tests/test_decisionpoint_conformance.py`. Determine whether it uses fixture globbing or an explicit list.
2. If globbing: no code change is strictly necessary, but add an assertion that the count of valid/invalid fixtures is what we expect (e.g., `assert len(valid_fixtures) >= 10`, `assert len(invalid_fixtures) >= 5`) to catch accidental deletions.
3. If explicit list: add each new fixture path to the appropriate list.
4. Add a parameterized test `test_v1_invalid_fixture_fails_schema` that iterates each invalid fixture and asserts `jsonschema.Draft202012Validator(schema).is_valid(fixture) is False` (or appropriate Pydantic rejection).
5. Add a parameterized test `test_v1_valid_fixture_passes_both_schema_and_pydantic` that iterates each valid fixture and asserts both validations pass.
6. For the cross-field invalid fixture (`v1_resolved_interview_deferred_with_final_answer.json`), the JSON Schema may or may not reject it (depends on whether the `if/then` constraints emitted by Pydantic are enforced by `jsonschema`). Always assert the Pydantic model_validate path rejects it. Add a specific test for this case.

**Files.** `tests/test_decisionpoint_conformance.py` (extend ~100 lines)

**Validation.**
- [ ] `pytest tests/test_decisionpoint_conformance.py -q` green.
- [ ] Deleting any fixture under `conformance/fixtures/decisionpoint/valid/` causes a test failure.
- [ ] Converting a valid fixture to invalid JSON causes a test failure.

## Risks

- **Envelope shape drift.** The test infers or hard-codes the event envelope shape. Fixtures must match the exact shape (including required fields like `lamport` and `actor_id`). Copy from an existing fixture verbatim for envelope; only the `payload` changes.
- **ParticipantIdentity fixture envelope.** If the conformance test doesn't support non-event fixtures, skip `v1_participant_identity_with_external_refs.json` or wrap it in an event envelope (e.g., embed it in a `ParticipantJoined` envelope). Preserve the spirit of FR-013 validation either way.
- **Cross-field rule not emitted by JSON Schema.** If Pydantic doesn't emit the `if/then/else` for cross-field Resolved constraints, the JSON Schema path may not catch the `deferred_with_final_answer` case. That's OK — the Pydantic path must still reject it, and the test must assert the rejection happens somewhere.

## Reviewer guidance

- Open each invalid fixture and verify the failure mode is precisely what its name claims.
- Verify every valid fixture uses timestamps in strict ISO-8601 form with timezone (no `Z` suffix if the rest of the repo uses `+00:00`, or vice versa — be consistent).
- Verify `package_data` globs in `pyproject.toml` already cover `conformance/fixtures/decisionpoint/valid/*.json` and `invalid/*.json` (they do — no change needed).
