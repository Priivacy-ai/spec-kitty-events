# WP04 — JSON schema regeneration

## Objective

Regenerate all committed DecisionPoint JSON Schemas from the new 4.0.0 Pydantic models, ship new shared-model schemas, and keep the schema-drift integration test zero-diff.

## Context

The repo convention (constraint C-006) is that Pydantic models are the source of truth and JSON Schemas are regenerated. `src/spec_kitty_events/schemas/generate.py` already orchestrates generation for 3.x models. V1 adds several new models and changes three existing payload shapes into discriminated unions.

See:
- `data-model.md` §7 — enumerated schema files.
- `research.md` R-9 — schema generation direction.
- `kitty-specs/.../contracts/*.schema.json` — planning-phase sketches that serve as the target shape for the generator's output (modulo Pydantic's emitted structure).

## Risks

- **Discriminator emission in Pydantic v2.** Pydantic emits the discriminator via `discriminator` keyword on the union `Field`. Confirm the JSON Schema output includes either a `discriminator` property (OpenAPI-style) or `oneOf` with per-variant const on the discriminator field. Either shape is acceptable as long as it's deterministic and validators accept it.
- **Schema drift on Pydantic version bump.** Pin the generator to the Pydantic version declared in `pyproject.toml` so CI doesn't drift on incidental upstream changes.
- **Shared-schema `$ref` emission.** Pydantic may inline `ParticipantIdentity` into every payload that references it. If the repo convention is to use `$ref` instead, adjust the generator accordingly — consult one existing payload (`participant_joined_payload.schema.json`) for the prevailing convention.
