# Work Packages: Mission Contract Cutover

**Inputs**: Design documents from `kitty-specs/014-mission-contract-cutover/`
**Prerequisites**: `plan.md` and `spec.md` required; `research.md`, `data-model.md`, `quickstart.md`, and `contracts/` used for implementation detail and validation guidance.

**Tests**: Included. This is a breaking contract release, and the spec explicitly requires conformance validation, replay validation, and the repository test suite to pass.

**Organization**: Fine-grained subtasks (`Txxx`) roll up into work packages (`WPxx`). Each work package is independently executable inside its owned files and sized to keep the prompt readable and reviewable.

**Prompt Files**: Each work package references a matching prompt file in `tasks/` generated by `/spec-kitty.tasks`.

## Subtask Format: `[Txxx] [P?] Description`

- **[P]** indicates the subtask can proceed in parallel with siblings inside the same WP because it targets a separate file or concern.
- Every subtask names concrete files or directories so ownership remains unambiguous.

---

## Work Package WP01: Cutover Artifact & Envelope Foundation (Priority: P0)

**Goal**: Establish the one authoritative cutover artifact, its helper semantics, and the canonical `Event` envelope split between `build_id` and `node_id`.
**Independent Test**: Artifact helper logic deterministically classifies canonical versus pre-cutover payloads, and the `Event` model requires `build_id` while preserving `node_id` Lamport semantics.
**Prompt**: `tasks/WP01-cutover-artifact-and-envelope-foundation.md`
**Requirement Refs**: FR-005, FR-006, FR-012, NFR-003, NFR-004, C-002
**Estimated Prompt Size**: ~330 lines

### Included Subtasks

- [ ] T001 Evaluate the existing release-authority surfaces and place the single authoritative cutover artifact under packaged source control in `src/spec_kitty_events/`
- [ ] T002 Implement the artifact interpreter/helper logic that enforces `signal_field_name`, `signal_location`, `cutover_contract_version`, `accepted_major`, and forbidden legacy surface semantics
- [ ] T003 Extend `src/spec_kitty_events/models.py` so canonical events require `build_id` and clearly preserve `node_id` as causal ordering identity only
- [ ] T004 Add or update unit and integration tests for artifact-driven fail-closed evaluation and the `build_id`/`node_id` envelope split

### Implementation Sketch

1. Inspect whether an existing packaged manifest can become the authoritative artifact without ambiguity; otherwise add a dedicated cutover artifact module or packaged JSON artifact.
2. Encode the exact on-wire signal binding and accepted-major policy in helper code that other repo-local validators can call.
3. Update the `Event` model and any helper constructors or serialization paths that need the new envelope field.
4. Lock the foundation down with direct tests before downstream contract families build on it.

### Parallel Opportunities

- T001 can complete first, then T002 and T003 can proceed in parallel once the artifact location decision is made.
- T004 can split between unit coverage for helper semantics and integration coverage for event creation paths.

### Dependencies

- None. This is the starting package and establishes the artifact and envelope contract that later WPs rely on.

### Risks & Mitigations

- Risk: an existing manifest looks reusable but cannot encode the full gate cleanly. Mitigation: promote reuse only if it can encode the exact signal binding, accepted major, and forbidden surfaces with no side rules.
- Risk: `build_id` breaks event construction across the repo. Mitigation: make all downstream fixture and test updates explicitly depend on this foundation WP.

## Work Package WP02: Lifecycle & Mission-Next Contract Cutover (Priority: P0)

**Goal**: Canonicalize lifecycle and mission-next contracts: `MissionCreated`/`MissionClosed`, lifecycle-only `MissionCompleted`, runtime-only `MissionRunCompleted`, and `mission_type` instead of `mission_key`.
**Independent Test**: Lifecycle and mission-next reducers accept the new canonical event names and fields, reject the runtime `MissionCompleted` alias path, and preserve one meaning per event name.
**Prompt**: `tasks/WP02-lifecycle-and-runtime-contract-cutover.md`
**Requirement Refs**: FR-001, FR-002, FR-003, FR-004, FR-008, NFR-002, C-002
**Estimated Prompt Size**: ~360 lines

### Included Subtasks

- [ ] T005 Add canonical lifecycle catalog event constants and typed payloads for `MissionCreated` and `MissionClosed` in `src/spec_kitty_events/lifecycle.py`
- [ ] T006 Keep `MissionCompleted` lifecycle-only by removing any catalog overloading and updating lifecycle reducer invariants and tests
- [ ] T007 Rename runtime mission workflow/template fields from `mission_key` to `mission_type` in `src/spec_kitty_events/mission_next.py` and its output models
- [ ] T008 Remove the `MissionCompleted` to `MissionRunCompleted` alias normalization path from `mission_next` and update reducer, replay, and determinism expectations

### Implementation Sketch

1. Introduce the canonical lifecycle catalog events and payload types.
2. Rework lifecycle reduction so `MissionCompleted` only represents lifecycle completion semantics.
3. Update mission-next runtime payloads and reducers to use `mission_type` and no runtime aliasing.
4. Refresh direct reducer and property coverage for the new semantics.

### Parallel Opportunities

- T005/T006 are tightly coupled in `lifecycle.py`.
- T007/T008 can proceed in parallel once the lifecycle event naming decisions are fixed because they stay inside `mission_next.py` and its tests.

### Dependencies

- Depends on WP01 because mission-next and lifecycle fixtures will consume the authoritative cutover artifact and the canonical on-wire signal defined there.

### Risks & Mitigations

- Risk: lifecycle and runtime completion semantics regress into ambiguous validation behavior. Mitigation: add explicit negative tests proving `MissionCompleted` is not accepted as a runtime alias.
- Risk: field renames break reducer state shape. Mitigation: update reducer output tests alongside payload models rather than leaving the shape drift for schema regeneration.

- [ ] T011 Update dossier and decisionpoint reducer, unit, and determinism tests for the canonical mission taxonomy
- [ ] T012 Preserve team-scoped `Project` semantics as documented contract meaning only and ensure these modules do not introduce runtime compatibility bridges

### Implementation Sketch

1. Update dossier terminology end-to-end, including namespace or output structures that surface mission identity.
2. Mirror the taxonomy cutover in decisionpoint payloads and reduced state.
3. Refresh reducer and property tests to lock the new field names and semantics.
4. Review touched docs/comments/examples in these modules to ensure project scoping remains descriptive, not enforced here.

### Parallel Opportunities

- T011 can split by test file once the field renames are in place.

### Dependencies

### Risks & Mitigations

## Work Package WP05: Schemas, Conformance & Replay Gates (Priority: P0)

**Goal**: Regenerate canonical schemas and fixtures, wire artifact-driven validation into conformance tooling, and prove fail-closed rejection across valid, invalid, and replay paths.
**Independent Test**: Schema generation, conformance validation, replay validation, and schema-drift checks all pass with the cutover artifact as the single compatibility source.
**Prompt**: `tasks/WP05-schemas-conformance-and-replay-gates.md`
**Requirement Refs**: FR-003, FR-005, FR-007, FR-008, FR-010, FR-011, FR-012, NFR-001, NFR-002, NFR-003, NFR-004, NFR-005, C-005
**Estimated Prompt Size**: ~430 lines

### Included Subtasks

- [ ] T017 Update `src/spec_kitty_events/schemas/generate.py` and schema loader surfaces for the cutover artifact, envelope changes, and renamed mission-domain payloads
- [ ] T018 Regenerate committed JSON schemas, including event envelope, lifecycle/runtime, projection, and cutover artifact schemas in `src/spec_kitty_events/schemas/`
- [ ] T019 Wire `src/spec_kitty_events/conformance/validators.py` and `loader.py` to enforce the artifact’s exact signal, `accepted_major`, and forbidden legacy surfaces
- [ ] T020 Rewrite `src/spec_kitty_events/conformance/fixtures/` and `manifest.json` so valid, invalid, and replay cases all use canonical mission/build terminology and explicit fail-closed cases
- [ ] T021 Update conformance, fixture, and schema-drift tests so the release bar proves artifact-driven validation and replay correctness end to end

### Implementation Sketch

1. Register the new or extended cutover artifact and renamed payloads in schema-generation code.
2. Regenerate committed schemas after source WPs land.
3. Route conformance validation through the artifact interpreter rather than duplicated constant lists.
4. Rewrite fixtures and replay streams to the canonical signal and terminology, including negative cases for missing signal, wrong major, and forbidden legacy surfaces.
5. Update conformance and drift tests to make these checks required.

### Parallel Opportunities

- T017 must land before T018.
- T019 and T020 can proceed in parallel once the artifact shape is stable.
- T021 can begin after initial fixture rewrites exist and finish after final regeneration.

### Dependencies

- Depends on WP01, WP02, WP03, and WP04 because schemas and fixtures must reflect the final canonical source models across all touched domains.

### Risks & Mitigations

- Risk: fixture rewrites miss one legacy surface and falsely pass partial validation. Mitigation: add explicit negative coverage for each forbidden surface class: keys, event names, aggregate names, and accepted-major mismatches.
- Risk: artifact semantics are duplicated in tests. Mitigation: have tests load and compare against the artifact rather than restating rule lists manually where possible.

## Dependency & Execution Summary

- **Sequence**: WP01 → (WP02, WP03, WP04 in parallel) → WP05 → WP06
- **Parallelization**: After the foundation WP lands, the lifecycle/runtime, dossier/decisionpoint, and mission-audit/status cutovers can proceed independently. Schema/fixture integration then consolidates the branch, and docs/versioning finish last.
- **MVP Scope**: WP01 is the foundation MVP because it defines the single artifact and envelope semantics that every later package depends on.
